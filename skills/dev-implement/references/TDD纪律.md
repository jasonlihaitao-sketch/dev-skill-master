# TDD 纪律与硬检查点

> 全程强制 Red → Green → Refactor。这是刚性规则，不可「灵活」。
> 测试是契约：它锁定 requirement 第 4 章验收标准与 task 第 3 验收点。

## 前后端适用度（重要）

- **后端代码：严格遵循红绿 TDD，无例外。**
  业务逻辑、数据访问、接口处理、算法等后端代码必须 Red→Green→Refactor，硬检查点全适用。
- **前端代码：尽量遵循，UI 视图层可酌情。**
  纯逻辑（hooks、工具函数、状态计算、数据转换）仍走红绿 TDD。
  视图/样式/纯展示组件（视觉渲染、布局、动效）难以单测时，可后置补充测试或用快照/视觉测试替代，
  但**必须在该任务的实施方案中注明「前端 UI，TDD 酌情」并说明替代验证方式**，不可静默跳过。

> 判断标准：能写断言的就必须先写测试；写不出断言的才允许酌情，且要记录原因。

## 循环

```
1. Red       写一个失败测试，表达当前验收点
2. 硬检查    运行测试 → 必须确认它失败（红）
3. Green     写最小实现让测试通过
4. 硬检查    运行测试 → 必须确认它通过（绿）
5. Refactor  重构实现（不改测试逻辑）
6. 硬检查    再运行测试 → 必须仍绿
```

## 硬检查点（违反即停）

这些检查之所以「硬」，是因为 TDD 的价值来自「测试先行」带来的设计反馈与可追溯性——一旦放宽，TDD 就退化成「事后补测」，失去意义。后端代码尤其不能松，因为业务逻辑的正确性全靠它保证。

1. **无失败测试不许写实现。** 先写测试，先看到红。红是你对需求的第一次精确表达，也是实现方向的锚点；先写实现再补测，测试只会描述你写了什么、而非需求要什么。
2. **必须实际运行测试确认红。** 不能凭假设跳过——测试可能误绿（说明需求已被覆盖或测试写错）。误绿的测试不构成约束，会给你「已覆盖」的假象。
3. **Green 后只许重构，不许改测试逻辑。** 测试锁定了行为契约，是需求验收标准的化身。要改行为，意味着需求变了，先改测试（回到 Red）再改实现，保证契约始终显式。
4. **一个任务的红绿循环不混入下一个任务。** 一个验收点一个循环，做完标完成再下一个。混着做会让失败原因无法定位，也破坏了「小步前进、随时可停」的安全感。
5. **每个验收点都要有对应测试。** task 第 3 验收点 ↔ requirement 第 4 章验收标准 ↔ 一个测试，三者可追溯。缺一环就可能在交付时漏掉某个需求点。

## 反模式（禁止）

- 先写实现再补测试（不是 TDD，是「测试后置」）
- 一次写多个测试再实现（破坏最小步进，难以定位）
- 为了让测试通过而修改测试（除非需求变了，且回到 Red）
- 跳过运行、假设红绿（必须实际跑）

## Red 阶段要点

- 测试名表达意图：`login_with_valid_credentials_returns_token`
- 一个测试只验一个行为
- 失败原因应是「行为未实现」，而非编译错或断言写错

## Green 阶段要点

- 写**最小**实现让测试通过，不要预判未来需求（YAGNI）
- 允许丑陋，留给 Refactor

## Refactor 阶段要点

- 行为不变，测试不变，只改结构
- 消除重复、改善命名、提取函数
- 每步重构后跑测试保持绿
