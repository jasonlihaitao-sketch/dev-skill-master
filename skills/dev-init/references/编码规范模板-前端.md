# 编码规范模板 — 前端（以 React / Vue / TypeScript 为主）

> 适用前提：本模板是「前端 React/Vue + TypeScript 栈」的规范参考。**仅当项目第 2 章技术栈判定为该前端栈时取用**；
> 若项目前端是其它栈（如原生 JS、非 TS、其它框架）且无对应模板，则**不套用本模板、不强制**，按项目自身约定执行。
> 跨端通用项（commit 规范、lint/format 工具位置）见 `project-context.md` 第 5 章，不在本文件重复。
>
> 规范形态：**强制条款 + 正例 + 反例 + 理由**。每条都应作为 dev-review 的检查项。
> React/Vue 二选一时，按项目实际框架取对应条目;差异大的章节分别给两框架正反例，共性章节给一种。

## 1. 命名

> 文件 / 组件 / 变量 / 常量 / 类型 / CSS 类。

**条款：组件文件大驼峰、组件名大驼峰名词;变量/函数小驼峰动词起;常量全大写下划线;类型/接口大驼峰。**

- ✅ 正例：组件文件 `UserCard.tsx` 导出 `UserCard`；变量 `handleSubmit`;常量 `MAX_COUNT`;类型 `type User = {...}`。
- ❌ 反例：组件文件 `userCard.tsx`（小写开头，与目录/导入混淆）;变量 `HandleSubmit`;常量 `maxCount`。
- 理由：命名一致降低认知成本;大小写区分组件与普通变量是 React/Vue 社区共识。

## 2. 组件结构【核心章节 · 深度】

> 组件拆分粒度与组织。React 用函数组件 + Hooks，Vue 用 `<script setup>` + 组合式 API。

### 2.1 单文件单组件，职责单一

**条款：一个组件文件只导出一个主组件;组件行数 ≤ 200 行（超了拆分）;容器组件（拉数据）与展示组件（纯渲染）分离。**

- ✅ 正例（React）：
  ```tsx
  // UserListPage.tsx —— 容器：拉数据、编排
  export function UserListPage() {
    const { data, isLoading } = useUsers();
    if (isLoading) return <Spinner />;
    return <UserList users={data} />;
  }
  // UserList.tsx —— 展示：纯渲染，props 驱动
  export function UserList({ users }: { users: User[] }) {
    return <ul>{users.map(u => <li key={u.id}>{u.name}</li>)}</ul>;
  }
  ```
- ✅ 正例（Vue）：
  ```vue
  <!-- UserListPage.vue -->
  <script setup lang="ts">
  const { data: users, isLoading } = useUsers();
  </script>
  <template><UserList v-if="!isLoading" :users="users" /></template>
  ```
- ❌ 反例：一个 `UserList` 组件里既 fetch 数据、又渲染、又处理表单提交，500 行。
- 理由：容器/展示分离让展示组件可纯渲染测试、可复用;单职责便于维护。

### 2.2 复杂逻辑抽 Hook / 组合式函数

**条款：组件内复杂副作用、状态逻辑抽成自定义 Hook（React `useXxx`）/ 组合式函数（Vue `useXxx`），组件只做编排。**

- ✅ 正例（React）：复杂表单逻辑抽 `useUserForm()`，组件只 `const { values, errors, submit } = useUserForm()`。
- ✅ 正例（Vue）：抽 `useUserForm()` 组合式函数，`<script setup>` 调用。
- ❌ 反例：表单校验、提交、状态全堆在组件 `useEffect`/`watch` 里，组件臃肿难测。
- 理由：Hook/组合式函数可独立单测、可复用;组件保持薄。

### 2.3 props 必须类型化，禁止 any

**条款：props 用类型/接口显式定义;禁止 `any`，必须 `unknown` + 类型收窄或显式类型。**

- ✅ 正例（React）：`function UserCard({ user }: { user: User })`。
- ✅ 正例（Vue）：`defineProps<{ user: User }>()` 或 `withDefaults(defineProps<UserProps>(), {...})`。
- ❌ 反例：`defineProps()` 不带类型（运行时校验缺失）;React props 标 `any`。
- 理由：类型化是 TS 核心价值;`any` 关闭类型检查等于退回 JS。

## 3. 状态管理【核心章节 · 深度】

> 本地 / 全局 / 服务端状态。能局部就不上全局。

### 3.1 状态就近原则，禁止滥用全局

**条款：状态优先局部（组件内）;只有跨多组件共享才上全局 store;服务端数据用 React Query/SWR 等而非手塞 store。**

- ✅ 正例（React）：表单临时态用 `useState`;用户登录态用 Zustand store;列表数据用 `useQuery`。
- ✅ 正例（Vue）：`ref` 局部状态;Pinia store 全局;服务端数据用 VueQuery。
- ❌ 反例：一个输入框的值也塞进全局 Redux/Pinia;接口返回数据手动存 store 再手动同步失效。
- 理由：全局状态增加耦合与调试成本;服务端状态有专门的缓存/失效机制，不该用通用 store 管。

### 3.2 服务端状态必须管缓存与失效

**条款：接口数据用 React Query/SWR/VueQuery，按 queryKey 管缓存与失效;写操作后显式 invalidate 相关 query。**

- ✅ 正例（React）：
  ```tsx
  const { data } = useQuery({ queryKey: ['users'], queryFn: fetchUsers });
  const mutation = useMutation({
    mutationFn: createUser,
    onSuccess: () => queryClient.invalidateQueries({ queryKey: ['users'] }),
  });
  ```
- ❌ 反例：`useEffect` 里 fetch + 手动 `setData`，无缓存无失效，重复请求、数据不一致。
- 理由：手管服务端状态易出竞态、重复请求、脏数据;专用库已解决。

### 3.3 全局 store 类型化、单一职责

**条款：全局 store 切片按业务域划分，state/actions 类型化;禁止一个巨型 store 装所有。**

- ✅ 正例：Zustand `useAuthStore`、`useCartStore` 分域;Pinia `useAuthStore`、`useCartStore` 分模块。
- ❌ 反例：一个 `useGlobalStore` 塞用户、购物车、UI 状态，改一处影响全局。
- 理由：分域 store 降低耦合、便于测试与 tree-shaking。

## 4. 类型（TypeScript）【核心章节 · 深度】

> 严格模式 + 类型来源 + 禁 any。

### 4.1 开启 strict，禁止 any

**条款：`tsconfig` 开 `strict: true`;禁止 `any`，必须 `unknown` + 收窄或显式类型;禁止 `@ts-ignore` 无注释。**

- ✅ 正例：`function parse(input: unknown): User { if (isUser(input)) return input; throw ... }`
- ❌ 反例：`function parse(input: any): any`;`// @ts-ignore` 无说明。
- 理由：`any` 关闭类型检查;`unknown` 强制收窄更安全。

### 4.2 API 响应类型必须定义且与后端契约对齐

**条款：接口响应必须有 TS 类型;理想从后端 schema（OpenAPI）生成，手写则集中放 `types/`，禁止内联散落。**

- ✅ 正例：`apiClient.get<User>('/users/1')` 返回 `Promise<User>`;`User` 定义在 `types/user.ts`。
- ❌ 反例：`apiClient.get('/users/1')` 返回 `any`;每个调用点重定义 `User` 类型。
- 理由：类型与契约对齐防字段漂移;集中定义避免重复与不一致。

### 4.3 共享类型集中，禁止重复定义

**条款：跨组件复用的类型放 `types/`，单一来源;组件私有类型可就地定义。**

- ✅ 正例：`types/user.ts` 导出 `User`，多处引用。
- ❌ 反例：A 组件定义 `User {id, name}`，B 组件又定义 `User {id, name, email}`，不一致。
- 理由：单一来源防漂移;改一处全更新。

## 5. 数据获取与接口【核心章节 · 深度】

> 统一请求层 + 错误处理 + 环境变量。

### 5.1 必须统一封装请求客户端

**条款：禁止散落 `fetch`/`axios` 调用;统一封装客户端，集中 baseURL、拦截器、错误处理、token 注入。**

- ✅ 正例：
  ```ts
  // api/client.ts
  export const apiClient = axios.create({ baseURL: import.meta.env.VITE_API_BASE });
  apiClient.interceptors.response.use(r => r.data, err => { if (err.status === 401) logout(); return Promise.reject(err); });
  // 业务：apiClient.get<User>('/users/1')
  ```
- ❌ 反例：每个组件里 `fetch('/api/users').then(...)`，baseURL、token、错误处理各写一遍。
- 理由：统一层保证一致性、可维护;拦截器集中处理鉴权与错误。

### 5.2 loading/error 状态必须显式管理

**条款：异步必须有 loading、error、success 三态显式处理;禁止只有 success 分支，loading/error 静默。**

- ✅ 正例：`const { data, isLoading, error } = useQuery(...)`;UI 三态分别渲染 Spinner/Error/Data。
- ❌ 反例：`fetch().then(setData)`，加载中无反馈、失败无提示，用户面对空白。
- 理由：三态缺失是体验与可用性 bug 高发区。

### 5.3 敏感配置走环境变量，禁止硬编码

**条款：baseURL、第三方 key 走 `import.meta.env`/`process.env`，前缀 `VITE_`/`NEXT_PUBLIC_`;禁止硬编码在源码;真正敏感的密钥不放前端。**

- ✅ 正例：`baseURL: import.meta.env.VITE_API_BASE`。
- ❌ 反例：`axios.create({ baseURL: 'https://prod.api.com' })` 硬编码;前端代码里塞服务端密钥。
- 理由：硬编码无法区分环境;前端可被反编译，敏感密钥必须放后端。

## 6. 样式（适度）

> 方案选择与约定。

**条款：选定一种样式方案（Tailwind/CSS Modules/styled-components/Vue `<style scoped>`），项目内统一;禁止内联样式堆积;主题色走 token/CSS 变量不硬编码。**

- ✅ 正例：Tailwind 统一用 `bg-primary` token;Vue `<style scoped>` 隔离。
- ❌ 反例：一半组件 `style={{color:'#fff'}}` 内联，一半用 class;主题色到处 `#3b82f6` 硬编码。
- 理由：方案统一降低维护成本;token 化便于换肤与一致性。

## 7. 可访问性 a11y（适度）

> 语义化与无障碍。

**条款：交互元素用语义化标签（`<button>` 不用 `<div onClick>`）;表单配 `label`;图片 `alt`;可交互元素可键盘聚焦。**

- ✅ 正例：`<button aria-label="关闭" onClick={onClose}>×</button>`;`<label htmlFor="email">邮箱</label><input id="email" />`。
- ❌ 反例：`<div onClick={submit}>提交</div>`（不可键盘聚焦、无语义）;图片 `<img />` 无 `alt`。
- 理由：语义化利于 SEO、屏幕阅读器、键盘用户;基础 a11y 是合规底线。

## 8. 测试约定【核心章节 · 深度】

> 与 dev-implement TDD 对齐（前端 UI 视图层可酌情，须注明替代验证）。

### 8.1 纯函数/Hook 必测，UI 视图可酌情

**条款：纯函数、工具函数、自定义 Hook/组合式函数必须单测;UI 组件测行为不测实现;纯 UI 视图层难测可酌情，但须注明「前端 UI，TDD 酌情」并给替代验证（手动走查清单）。**

- ✅ 正例（React）：
  ```tsx
  // useUserForm.test.tsx
  test('should validate email', () => {
    const { result } = renderHook(() => useUserForm());
    act(() => result.current.setEmail('bad'));
    expect(result.current.errors.email).toBeDefined();
  });
  // UserCard.test.tsx —— 测行为
  test('renders user name', () => {
    render(<UserCard user={{ id: '1', name: 'Tom' }} />);
    expect(screen.getByText('Tom')).toBeInTheDocument();
  });
  ```
- ❌ 反例：只测「组件能渲染」不断言行为;测组件内部 state 实现细节（`expect(wrapper.state())`）。
- 理由：测行为（用户视角）稳定，测实现（内部 state）脆弱;纯逻辑必测是质量底线。

### 8.2 E2E 覆盖关键用户流

**条款：关键用户流（登录、下单、支付）用 Playwright/Cypress 覆盖;禁止 E2E 测琐碎细节。**

- ✅ 正例：Playwright 测「登录→加购→下单→支付成功」主流程。
- ❌ 反例：E2E 测按钮颜色、单个组件渲染。
- 理由：E2E 成本高，应聚焦端到端价值流;细节交单测。

## 9. 性能（适度）

> 渲染与加载。

**条款：路由懒加载（`React.lazy`/Vue 异步组件）;大依赖动态 import;避免无意义重渲染（按需 `memo`/`useMemo`/`computed`）。**

- ✅ 正例：`const Admin = React.lazy(() => import('./Admin'))`;列表项 `memo` 防重排。
- ❌ 反例：所有组件都 `memo`（过度优化）;路由全量打包首屏加载慢。
- 理由：懒加载降首屏成本;`memo` 按需而非滥用，滥用反增开销。

---

## 10. 安全（适度）

> 与 dev-review 安全维度对齐。

**条款：禁止 `dangerouslySetInnerHTML`/`v-html` 直渲用户输入（XSS）;token 存储用 httpOnly cookie 或短时内存;定期 `npm audit`;锁定依赖版本。**

- ✅ 正例：富文本用 DOMPurify 白名单清洗后再渲染;token 走 httpOnly cookie。
- ❌ 反例：`<div dangerouslySetInnerHTML={{__html: userContent}} />`（用户输入直渲 = XSS）;token 存 localStorage 长期有效。
- 理由：XSS 是前端头号漏洞;token 存储方式决定被盗风险。
