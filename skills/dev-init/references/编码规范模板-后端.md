# 编码规范模板 — 后端（以 Java 为主）

> 适用前提：本模板是「后端 Java 栈」的规范参考。**仅当项目第 2 章技术栈判定为 Java 后端时取用**；
> 若项目后端是 Python/Go/Node 等非 Java 栈且无对应模板，则**不套用本模板、不强制**，按项目自身约定执行。
> 跨端通用项（commit 规范、lint/format 工具位置）见 `project-context.md` 第 5 章，不在本文件重复。
>
> 用法：逐字段填充，无内容写「待定」并列入待澄清；范例仅供示范，按项目实际裁剪。

## 1. 命名

> 包 / 类 / 方法 / 变量 / 常量 / 枚举命名约定。

- **包名**：（全小写，倒序域名，如 `com.example.user`）
- **类名**：（大驼峰，名词，如 `UserService`；接口不加 `I` 前缀）
- **方法名**：（小驼峰，动词起，如 `findById`）
- **常量**：（全大写下划线，`MAX_RETRY`）
- **枚举**：（枚举值全大写下划线）
- 范例：Controller 只做参数校验与编排，类名 `XxxController` / `XxxService` / `XxxRepository` 分层命名。

## 2. 分层与包结构

> 分层职责与目录组织。

- **分层**：（Controller / Service / Repository / Domain / Infra，按项目实际）
- **职责边界**：（如 Controller 不含业务逻辑；Service 不直接返回 HTTP 状态）
- **包组织**：（按业务模块 vs 按技术层，二选一并说明）
- 范例：`com.example.user.controller` / `.service` / `.repository` / `.domain`，跨模块依赖只走接口。

## 3. 错误处理

> 异常体系与处理策略。

- **异常分类**：（业务异常 `BizException` / 系统异常 / 校验异常，是否自定义基类）
- **抛出与捕获边界**：（Service 抛业务异常，统一在 Controller 层或全局 `@RestControllerAdvice` 捕获转换）
- **错误码**：（是否有统一错误码体系，枚举定义位置）
- **不吞异常**：（禁止空 catch；必须记录或向上抛）
- 范例：业务校验失败抛 `BizException(ErrorCode.USER_NOT_FOUND)`，全局处理器转 `{code, message}` 响应。

## 4. 日志

> 日志级别、内容、脱敏。

- **级别约定**：（ERROR 异常堆栈 / WARN 可恢复异常 / INFO 关键业务节点 / DEBUG 调试）
- **内容要求**：（带 traceId / 关键业务 ID；禁止打印密码、token、身份证等敏感信息）
- **框架与配置**：（如 SLF4J + Logback，配置位置）
- 范例：`log.info("createOrder ok, userId={}, orderId={}", userId, orderId);` 用占位符不用字符串拼接。

## 5. API 设计

> REST / RPC 接口规范。

- **URL 风格**：（RESTful，复数名词，`/api/v1/users/{id}`）
- **请求/响应**：（统一包装 `{code, message, data}` 与否；DTO 与领域对象分离）
- **版本**：（URL 版本 `/v1` 或 Header 版本）
- **幂等与并发**：（写接口是否要求幂等键；并发更新用乐观锁/版本号）
- 范例：POST `/api/v1/orders` 创建，幂等键 `Idempotency-Key` Header；分页用 `page`/`size`，响应含 `total`。

## 6. 并发与事务

> 事务边界、并发安全。

- **事务**：（`@Transactional` 标注在 Service 层；只读查询加 `readOnly=true`；事务方法不跨 HTTP 调用）
- **并发**：（共享可变状态用并发容器/锁；避免在 Service 持有可变字段）
- 范例：转账 Service 方法标 `@Transactional`，先查后改；并发扣减库存用乐观锁版本号。

## 7. 数据访问

> Repository / SQL / 缓存。

- **ORM**：（JPA / MyBatis，N+1 防范）
- **SQL 约定**：（禁止 `SELECT *`；分页必带；慢查询阈值）
- **缓存**：（缓存键命名、失效策略、防穿透/击穿）
- 范例：Repository 返回领域对象或 Optional，不返回 Map；查询用 `@Query` 显式列。

## 8. 测试约定

> 单测/集成测试规范，与 dev-implement TDD 对齐。

- **命名**：（`should_xxx_when_yyy` 或 `methodName_条件_预期`）
- **结构**：（Given-When-Then 或 Arrange-Act-Assert）
- **边界**：（Service 单测用 Mock；Repository 走集成测试；不测私有方法）
- **覆盖率门槛**：（如行覆盖 ≥ 80%，核心业务 ≥ 90%）
- 范例：`UserServiceTest.should_throw_when_user_not_found`，用 Mockito mock Repository。

## 9. 安全

> 与 dev-review 安全维度对齐。

- **输入校验**：（`@Valid` + Bean Validation；白名单校验）
- **认证授权**：（鉴权注解位置；越权检查在 Service 层）
- **敏感数据**：（密码 BCrypt；配置项不入库不入日志）
- 范例：所有写接口校验入参；查询他人数据必须校验归属权。
