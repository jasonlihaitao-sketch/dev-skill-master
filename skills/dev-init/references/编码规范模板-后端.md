# 编码规范模板 — 后端（以 Java 为主）

> 适用前提：本模板是「后端 Java 栈」的规范参考。**仅当项目第 2 章技术栈判定为 Java 后端时取用**；
> 若项目后端是 Python/Go/Node 等非 Java 栈且无对应模板，则**不套用本模板、不强制**，按项目自身约定执行。
> 跨端通用项（commit 规范、lint/format 工具位置）见 `project-context.md` 第 5 章，不在本文件重复。
>
> 规范形态：**强制条款 + 正例 + 反例 + 理由**。每条都应作为 dev-review 的检查项。正例/反例仅供示范，类名/包名按项目实际调整。

---

## 1. 命名

> 包 / 类 / 方法 / 变量 / 常量 / 枚举命名约定。

- **包名**：全小写，倒序域名，如 `com.example.user`。
- **类名**：大驼峰，名词，如 `UserService`；接口不加 `I` 前缀。
- **方法名**：小驼峰，动词起，如 `findById`。
- **常量**：全大写下划线，如 `MAX_RETRY`。
- **枚举值**：全大写下划线。

**条款：Controller/Service/Repository 分层命名，类名须体现职责，禁止 `Util`/`Helper`/`Manager` 滥用。**

- ✅ 正例：`UserController`（编排）、`UserService`（业务）、`UserRepository`（数据访问）、`User`（领域对象）。
- ❌ 反例：`UserManager`（职责不清）、`UserUtil`（什么都往里塞）、`UserData`（分不清是领域还是 DTO）。
- 理由：分层命名让职责一眼可辨，review 与排错时能按类名定位层级。

## 2. 分层与包结构

> 分层职责与目录组织。

- **分层**：Controller / Service / Repository / Domain / Infra。
- **职责边界**：Controller 只做参数校验与编排，不含业务逻辑；Service 不直接返回 HTTP 状态。
- **包组织**：按业务模块组织（推荐），跨模块依赖只走接口。

**条款：Controller 不含业务逻辑；Service 不直接操作 HTTP 响应；Repository 只做数据访问。**

- ✅ 正例：
  ```java
  @RestController
  public class UserController {
      private final UserService userService;
      @PostMapping("/users")
      public Result<UserVO> create(@Valid @RequestBody CreateUserRequest req) {
          return Result.success(userService.create(req));
      }
  }
  // Service 负责业务，返回领域对象/VO；Repository 负责持久化
  ```
- ❌ 反例：Controller 里写 `if (user == null) throw ...; user.setStatus(...); repository.save(user);` —— 业务逻辑泄漏到 Controller。
- 理由：分层隔离让每层可测、可替换；逻辑散落 Controller 会无法单测、难维护。

## 3. 异常处理

> 自定义异常体系 + 自定义异常码 + 清晰的异常码目录。

### 3.1 必须自定义业务异常体系，禁止抛裸异常

**条款：业务异常必须继承统一基类；禁止直接抛 `RuntimeException`/`Exception`/`NullPointerException`。**

- ✅ 正例：
  ```java
  public class BizException extends RuntimeException {
      private final ErrorCode errorCode;
      public BizException(ErrorCode errorCode) { super(errorCode.getMessage()); this.errorCode = errorCode; }
      public ErrorCode getErrorCode() { return errorCode; }
  }
  // 使用
  if (user == null) throw new BizException(ErrorCode.USER_NOT_FOUND);
  ```
- ❌ 反例：`throw new RuntimeException("用户不存在");` 或 `throw new NullPointerException();`
- 理由：裸异常无法区分业务/系统、调用方无法据码处理、message 文案变更即破坏契约。

### 3.2 必须自定义异常码，禁止用 message 传达业务错误

**条款：异常携带错误码而非靠 message 区分；调用方据码处理，不解析文案。**

- ✅ 正例：`throw new BizException(ErrorCode.USER_NOT_FOUND);` 前端据 `errorCode` 判断走什么交互。
- ❌ 反例：`throw new BizException("用户不存在")`，前端靠字符串 `"用户不存在"` 判断 —— 文案一改前端就崩。
- 理由：错误码是稳定契约，message 是给人看的可变文案。

### 3.3 必须有清晰的异常码目录（统一枚举 + 分段编号 + 文档）

**条款：所有错误码集中定义在统一枚举，按业务模块分段编号，每项带文档注释（码值、含义、触发场景）；禁止散落硬编码。**

- ✅ 正例：
  ```java
  public enum ErrorCode {
      /** 10000-19999 用户模块 */
      USER_NOT_FOUND(10001, "用户不存在", "按 id 查询用户时返回"),
      USER_PWD_ERROR(10002, "密码错误", "登录校验失败"),
      /** 20000-29999 订单模块 */
      ORDER_NOT_FOUND(20001, "订单不存在", "查询订单时返回"),
      ORDER_STOCK_NOT_ENOUGH(20002, "库存不足", "下单扣减库存失败");

      private final int code; private final String message; private final String scene;
      // 构造、getter 略
  }
  ```
- ❌ 反例：各 Service 里 `throw new BizException(50001, "xxx")` —— 错误码散落，冲突无控、无目录、无文档。
- 理由：集中目录让错误码可查、可审计、不冲突；分段编号为新增预留段位。

### 3.4 抛出与捕获边界

**条款：Service 抛业务异常；禁止在 Controller 层吞异常或拼错误码；未知异常由全局处理器转系统错误码。**

- ✅ 正例：
  ```java
  @RestControllerAdvice
  public class GlobalExceptionHandler {
      @ExceptionHandler(BizException.class)
      public Result<Void> handleBiz(BizException e) {
          return Result.fail(e.getErrorCode());
      }
      @ExceptionHandler(Exception.class)
      public Result<Void> handleSys(Exception e) {
          log.error("系统异常", e);  // 记录完整堆栈
          return Result.fail(ErrorCode.SYSTEM_ERROR);  // 不向前端泄露堆栈
      }
  }
  ```
- ❌ 反例：Controller 里 `try { ... } catch (Exception e) { return "error"; }` —— 吞异常、丢失码、无日志。
- 理由：统一捕获让响应一致、错误码可控；吞异常会让故障不可见。

### 3.5 禁止空 catch；敏感信息脱敏

**条款：catch 后必须记录或向上抛，禁止什么也不做；密码/token 等不进异常 message 与堆栈日志。**

- ✅ 正例：`catch (IOException e) { log.error("读取配置失败", e); throw new SysException(ErrorCode.CONFIG_READ_FAIL, e); }`
- ❌ 反例：`catch (Exception e) { }` 或 `catch (Exception e) { e.printStackTrace(); }`
- 理由：空 catch 掩盖故障；`printStackTrace` 不进日志框架、生产难定位。

## 4. 接口规范

> 入参/出参对象化 + 统一成功码与错误码 + 统一响应包装。

### 4.1 入参/出参必须用对象，禁止分散参数

**条款：接口入参、出参一律用对象（Request/Response DTO），禁止 5+ 个分散参数；超过 2 个参数强制封装。**

- ✅ 正例：
  ```java
  @PostMapping("/users")
  public Result<UserVO> create(@Valid @RequestBody CreateUserRequest req) { ... }

  public class CreateUserRequest {
      @NotBlank private String name;
      @NotNull @Email private String email;
      @Min(0) private Integer age;
      // getter/setter
  }
  ```
- ❌ 反例：`public Result<UserVO> create(String name, String email, Integer age, String phone, String addr, Integer role)`
- 理由：对象化便于校验（`@Valid`）、扩展（加字段不动签名）、文档化、前端对接；分散参数难维护、易传错位。

### 4.2 所有返回对象必须有统一的成功码与错误码

**条款：所有接口返回统一包装 `{code, message, data}`；成功用统一成功码（如 `0` 或 `200`），失败用错误码目录里的码；禁止裸返回领域对象、禁止无码响应。**

- ✅ 正例：
  ```java
  public class Result<T> {
      private int code; private String message; private T data;
      public static <T> Result<T> success(T data) { return new Result<>(0, "success", data); }
      public static <T> Result<T> fail(ErrorCode ec) { return new Result<>(ec.getCode(), ec.getMessage(), null); }
  }
  // Controller 一律返回 Result<T>
  @GetMapping("/users/{id}")
  public Result<UserVO> get(@PathVariable Long id) { return Result.success(userService.get(id)); }
  ```
- ❌ 反例：`public UserVO get(Long id)` 直接返回 VO —— 前端无法区分成功/失败、无统一码；或成功返回 `{code:0}`、失败返回 `{errCode:1001}` 两种结构。
- 理由：统一包装让前端解析逻辑唯一、错误处理走码；无码响应无法被统一拦截与监控。

### 4.3 URL 与版本

**条款：RESTful 复数名词 + URL 版本；DTO 与领域对象分离，禁止领域对象直接出接口。**

- ✅ 正例：`POST /api/v1/users`、`GET /api/v1/users/{id}`；返回 `UserVO` 而非 `User`。
- ❌ 反例：`POST /api/getUserById`（动词 URL）、直接返回带密码字段的 `User` 实体。
- 理由：版本便于演进；领域对象出接口会泄露内部结构、引发过度耦合。

## 5. SQL 规范

> 禁止动态 SQL / 字符串拼接，用固定常量 + 参数化。

### 5.1 禁止动态 SQL 与字符串拼接，必须参数化预编译

**条款：SQL 必须用固定常量 + 参数绑定（`?` 或 `#{}`），禁止运行时拼接表名/列名/条件；MyBatis 用 `#{}` 防 SQL 注入，`${}` 仅限排序字段等白名单且须校验。**

- ✅ 正例（MyBatis）：
  ```xml
  <select id="findById" resultType="User">
      SELECT id, name, email FROM user WHERE id = #{id}
  </select>
  ```
  JPA：`@Query("SELECT u FROM User u WHERE u.id = :id") User findById(@Param("id") Long id);`
- ❌ 反例：`"SELECT * FROM user WHERE name = '" + name + "'"` —— SQL 注入 + 动态拼接。
  或 MyBatis `WHERE name = '${name}'` —— `${}` 不参数化。
- 理由：参数化防 SQL 注入；固定 SQL 可被数据库预编译缓存、计划复用;拼接即注入风险。

### 5.2 禁止 SELECT *，列必须显式

**条款：查询列必须显式列出，禁止 `SELECT *`。**

- ✅ 正例：`SELECT id, name, email FROM user`
- ❌ 反例：`SELECT * FROM user`
- 理由：`*` 取多余列浪费带宽/内存、表结构变更隐式影响、无法用覆盖索引优化。

### 5.3 写操作必须指定列，禁止全表更新/删除

**条款：UPDATE/DELETE 必须带 WHERE 条件且条件列有索引；禁止无 WHERE 的写操作。**

- ✅ 正例：`UPDATE user SET status = 1 WHERE id = #{id}`
- ❌ 反例：`UPDATE user SET status = 1`（全表更新）、`DELETE FROM user`（清表）
- 理由：无条件写操作是灾难性事故;带索引 WHERE 防锁表。

### 5.4 禁止大事务 SQL，分页必带

**条款：查询大表必须分页（`LIMIT`/`OFFSET` 或 keyset），禁止一次性全量拉取。**

- ✅ 正例：`SELECT id, name FROM user WHERE status = 0 ORDER BY id LIMIT 20 OFFSET 0`
- ❌ 反例：`SELECT * FROM user`（全表加载到内存）
- 理由：全量拉取会 OOM、拖垮 DB；分页是基本安全门。

## 6. 定时任务

> `@Scheduled` / Quartz 等。

### 6.1 cron 用固定常量，禁止动态表达式

**条款：cron 表达式必须用固定常量或配置项，禁止运行时动态生成；需调频率走配置中心/`@Value`，不写死在注解字面量。**

- ✅ 正例：
  ```java
  @Scheduled(cron = "${task.sync.cron:0 0 2 * * ?}")  // 默认值兜底，可配置覆盖
  public void syncData() { ... }
  ```
- ❌ 反例：`@Scheduled(cron = buildCronFromDB())` —— 动态拼 cron，难审计、难排查。
- 理由：固定 cron 可见、可审计;动态表达式让调度不可预测。

### 6.2 任务必须幂等

**条款：定时任务必须幂等，重复执行不产生脏数据。**

- ✅ 正例：同步任务用「最后同步时间」或唯一键去重，重复执行只补差量。
- ❌ 反例：每次执行都 `INSERT` 不判重，节点重启重试产生重复数据。
- 理由：定时任务可能因重启/重试重复触发,幂等是安全底线。

### 6.3 分布式部署必须加分布式锁

**条款：多节点部署时，定时任务必须加分布式锁，确保全集群只有一个节点执行。**

- ✅ 正例：用 Redis 锁 / 数据库锁，`if (tryLock("task:sync")) { execute(); }`
- ❌ 反例：多节点都跑 `@Scheduled`，每个节点都执行一遍同步。
- 理由：多节点重复执行导致重复数据/重复调用下游、资源浪费。

### 6.4 异常必须捕获并记录，不能静默挂掉

**条款：任务体内必须 try-catch 记录异常，不能让异常抛出导致调度线程静默停止。**

- ✅ 正例：
  ```java
  @Scheduled(cron = "...")
  public void syncData() {
      try { doSync(); }
      catch (Exception e) { log.error("同步任务失败", e); }  // 记录但不中断调度
  }
  ```
- ❌ 反例：任务体直接 `doSync()` 不 catch —— 抛异常后某些调度器会停止该任务，后续不再执行。
- 理由：调度静默挂掉是隐蔽故障,捕获记录保证持续运行可监控。

### 6.5 长任务设超时

**条款：长任务必须有超时/并发控制，防单次执行卡死阻塞下次。**

- ✅ 正例：任务内对每批处理设超时；或用 `@Scheduled` 配合 `fixedDelay`（等上次完成再计时）。
- ❌ 反例：任务无超时，某次卡在远程调用，下次触发时重叠执行。
- 理由：无超时的长任务会积压、资源耗尽。

## 7. 线程池

> 显式 ThreadPoolExecutor + 队列按场景选 + 核心参数 + 拒绝策略 + 命名 + 监控 + 总量受限。

### 7.1 禁止用 Executors，必须显式 ThreadPoolExecutor

**条款：禁止 `Executors.newFixedThreadPool`/`newCachedThreadPool`/`newSingleThreadExecutor`，必须用 `ThreadPoolExecutor` 显式构造。**

- ✅ 正例：
  ```java
  ThreadPoolExecutor executor = new ThreadPoolExecutor(
      8, 16, 60L, TimeUnit.SECONDS,
      new LinkedBlockingQueue<>(1000),
      new ThreadFactoryBuilder().setNameFormat("order-pool-%d").build(),
      new ThreadPoolExecutor.CallerRunsPolicy());
  ```
- ❌ 反例：`ExecutorService es = Executors.newFixedThreadPool(10);`
- 理由：`newFixedThreadPool`/`newSingleThreadExecutor` 用无界 `LinkedBlockingQueue`，积压 OOM;`newCachedThreadPool` 最大线程无界，线程爆炸。显式构造才能控边界（阿里规约）。

### 7.2 队列按场景选，禁止无界队列

**条款：队列按场景选择，必须设容量（有界），禁止无界队列。**

| 场景 | 队列 | 说明 |
|---|---|---|
| 一般业务，可控积压 | `LinkedBlockingQueue`(指定容量) / `ArrayBlockingQueue` | 有界，防 OOM |
| 高吞吐、短任务、直接 handoff | `SynchronousQueue` | 不存储，直接交给线程；配合大 maxPool |
| 任务有优先级 | `PriorityBlockingQueue` | 按优先级调度 |

- ✅ 正例：`new LinkedBlockingQueue<>(1000)` —— 有界 1000。
- ❌ 反例：`new LinkedBlockingQueue<>()`（不传容量 = Integer.MAX_VALUE，无界 OOM）。
- 理由：无界队列积压导致 OOM;有界队列让拒绝策略生效、可感知过载。

### 7.3 核心参数（最大/最小性能指数）必须显式设定并给依据

**条款：`corePoolSize`/`maximumPoolSize`/队列容量/拒绝策略必须显式设定，并给出推导依据，禁止拍脑袋默认值。**

- **核心线程数 `corePoolSize`（最小性能指数）**：
  - CPU 密集型 ≈ `Ncpu + 1`（N=CPU 核心数）
  - IO 密集型 ≈ `2 * Ncpu`（或 `Ncpu / (1 - 阻断比)`）
- **最大线程数 `maximumPoolSize`（最大性能指数）**：峰值承载，按 `目标QPS × 平均任务耗时` 推导，留 buffer。
- **队列容量**：按积压容忍度（如允许积压 1 秒 = `QPS × 1s`）。
- **拒绝策略**：
  - `AbortPolicy`（默认）—— 抛异常可感知，关键业务用。
  - `CallerRunsPolicy` —— 回压降级，由提交线程自己跑，可接受降速时用。
  - `DiscardPolicy`/`DiscardOldestPolicy` —— 静默丢弃，慎用，须有监控。

- ✅ 正例：
  ```java
  // IO 密集型，8 核，core=16, max=32, 队列 500, 拒绝回压
  new ThreadPoolExecutor(16, 32, 60L, TimeUnit.SECONDS,
      new LinkedBlockingQueue<>(500),
      new ThreadFactoryBuilder().setNameFormat("biz-pool-%d").build(),
      new ThreadPoolExecutor.CallerRunsPolicy());
  ```
- ❌ 反例：`new ThreadPoolExecutor(0, Integer.MAX_VALUE, ...)` —— max 无界，线程爆炸。
- 理由：参数决定稳定性与性能；显式 + 依据让参数可审计、可调优。

### 7.4 线程池必须命名 + 暴露监控

**条款：线程必须通过 `ThreadFactory` 命名（`biz-pool-%d`），便于线程栈排查；线程池必须暴露监控（活跃数/队列积压/拒绝次数）。**

- ✅ 正例：`ThreadFactoryBuilder().setNameFormat("order-pool-%d")`；接入 Micrometer 暴露 `executor.active`/`queue.size`/`rejected`。
- ❌ 反例：默认 `pool-1-thread-1`，排查时不知是谁的池；无监控，过载了才发现。
- 理由：命名让 jstack 可定位;监控让过载可预警。

### 7.5 服务内线程池总量受限，集中管理

**条款：服务内线程池数量必须有上限并集中管理（统一配置/注入），禁止每个类随手 `new` 线程池。**

- ✅ 正例：线程池在配置类统一 `@Bean` 定义，业务类注入使用;全局线程池数量 ≤ 个位数。
- ❌ 反例：每个 Service 方法里 `new ThreadPoolExecutor(...)`，几十个池，线程数失控。
- 理由：线程池失控 = 线程爆炸、资源耗尽、调度混乱;集中管理才可控可监控。

## 8. 事务配置

> `@Transactional` 边界 + 传播 + 隔离 + 超时 + 只读 + 回滚 + 代理陷阱。

### 8.1 标注在 Service 层，禁止标 Controller/Repository

**条款：`@Transactional` 标注在 Service 层方法/类上，禁止标 Controller、Repository。**

- ✅ 正例：`@Service public class OrderService { @Transactional public void create(...) {} }`
- ❌ 反例：`@RestController @Transactional class OrderController` —— 事务边界错位。
- 理由：事务是业务一致性边界，应在 Service 层。

### 8.2 只读查询显式 readOnly，必须设超时

**条款：只读查询标 `readOnly = true`；所有事务必须设 `timeout`，防长事务锁表。**

- ✅ 正例：`@Transactional(readOnly = true, timeout = 30) public Order findById(Long id) {}`
- ❌ 反例：`@Transactional public Order findById(Long id) {}` —— 未声明只读、无超时。
- 理由：`readOnly` 提示优化、防误写;超时防长事务持锁。

### 8.3 明确传播行为与隔离级别，禁止无思考用默认

**条款：传播行为、隔离级别必须显式声明并说明理由;`REQUIRES_NEW` 用于独立事务（如日志不随主事务回滚）。**

- ✅ 正例：`@Transactional(propagation = Propagation.REQUIRES_NEW) public void saveLog() {}` —— 日志独立提交。
- ❌ 反例：默认 `REQUIRED` 但开发者不知含义，日志记在主事务内，主事务回滚日志也没了。
- 理由：传播/隔离影响一致性与并发行为,显式声明才可控。

### 8.4 checked 异常回滚必须显式 rollbackFor

**条款：默认只回滚 `RuntimeException`;checked 异常要回滚必须显式 `rollbackFor = Exception.class`。**

- ✅ 正例：`@Transactional(rollbackFor = Exception.class) public void upload(...) throws IOException {}`
- ❌ 反例：方法抛 `IOException` 但事务未回滚，数据不一致。
- 理由：Spring 默认不回滚 checked 异常,易导致部分提交。

### 8.5 事务内禁止耗时操作与跨 RPC/HTTP 调用

**条款：事务方法内禁止文件上传、远程调用、`Thread.sleep` 等耗时操作;事务不跨 RPC/HTTP 调用（先提交再调用）。**

- ✅ 正例：先 `@Transactional` 提交订单，再在事务外调用支付/通知。
- ❌ 反例：事务内 `httpClient.callPayApi()`，支付接口慢导致事务长时间持锁。
- 理由：事务持锁期间做慢操作 = 连接占用、锁等待、超时。

### 8.6 事务方法不可 private/final/self-invocation

**条款：`@Transactional` 方法不能是 `private`/`final`/`static`;同类内部直接调用事务方法不生效（Spring AOP 代理失效），须走代理或抽到另一个 Bean。**

- ✅ 正例：`orderService.create()` 由外部调用，代理生效;同类内调用用 `AopContext.currentProxy()` 或拆 Bean。
- ❌ 反例：`class OrderService { @Transactional public void a(){} public void b(){ this.a(); } }` —— `this.a()` 走原生对象，事务不生效。
- 理由：Spring 事务基于 AOP 代理，self-invocation 绕过代理，事务静默失效。

## 9. 数据库配置

> 连接池 + 超时 + 敏感配置 + ORM。

### 9.1 必须用连接池，禁止直连

**条款：必须用连接池（HikariCP/Druid），禁止裸 `DriverManager` 获取连接。**

- ✅ 正例：Spring Boot 默认 HikariCP，配 `spring.datasource.hikari.*`。
- ❌ 反例：`DriverManager.getConnection(url, user, pwd)` 每次新建连接。
- 理由：裸连接开销大、无法复用;连接池是基本工程实践。

### 9.2 连接池参数必须配上限

**条款：`maximumPoolSize`/`minimumIdle`/`connectionTimeout`/`idleTimeout` 必须显式配置，禁止用无界默认。**

- ✅ 正例（Hikari）：
  ```yaml
  spring:
    datasource:
      hikari:
        maximum-pool-size: 20
        minimum-idle: 5
        connection-timeout: 3000
        idle-timeout: 600000
        max-lifetime: 1800000
  ```
- ❌ 反例：用默认 `maximumPoolSize=10` 且不评估，高并发不够、低并发浪费。
- 理由：连接数影响 DB 与应用稳定性;按 `((核心数*2) + 有效磁盘数)` 或压测定。

### 9.3 必须设 SQL 超时

**条款：必须设查询超时/连接空闲超时，防慢查询拖垮连接池。**

- ✅ 正例：Hikari `connection-timeout` + JPA `@QueryHints`/MyBatis `timeout` 或 DB 层 `statement_timeout`。
- ❌ 反例：无超时，一条慢 SQL 占住连接，连接池耗尽全站不可用。
- 理由：慢查询是连接池耗尽主因;超时是兜底。

### 9.4 敏感配置不硬编码

**条款：数据库账号密码走环境变量/配置中心/Vault，禁止硬编码在代码或入库不入日志。**

- ✅ 正例：`spring.datasource.password: ${DB_PASSWORD}`（环境变量/配置中心注入）。
- ❌ 反例：`password: "root123"` 写死在 `application.yml` 提交到 Git。
- 理由：硬编码凭证 = 泄露风险、轮换困难。

### 9.5 ORM 选型与 N+1 防范

**条款：JPA 禁 N+1（用 `@EntityGraph`/fetch join）;MyBatis 列显式;禁止领域对象直接出接口。**

- ✅ 正例（JPA）：`@EntityGraph(attributePaths = "orders") List<User> findAll();` 一次查关联。
- ❌ 反例：查 N 个用户后循环查每个用户的订单 —— N+1，DB 被打爆。
- 理由：N+1 是性能头号杀手;显式列 + fetch join 是基本优化。

## 10. 日志

> 级别 + 内容 + 脱敏 + 框架。

**条款：用 SLF4J 占位符不用字符串拼接;ERROR 带堆栈;INFO 关键业务节点带 traceId/业务 ID;敏感信息脱敏。**

- ✅ 正例：`log.info("createOrder ok, userId={}, orderId={}", userId, orderId);` —— `log.error("支付失败", e)` 带异常对象打印堆栈。
- ❌ 反例：`log.info("createOrder ok " + userId)`（字符串拼接，未开日志也构造字符串）；`log.error("失败: " + e.getMessage())`（丢堆栈）；日志里打密码明文。
- 理由：占位符延迟构造、省性能;堆栈是定位关键;脱敏是合规底线。

## 11. 安全

> 与 dev-review 安全维度对齐。

**条款：写接口必须 `@Valid` 校验入参;查询他人数据必须校验归属权;密码 BCrypt;敏感数据不入日志。**

- ✅ 正例：`@PostMapping public Result<?> create(@Valid @RequestBody Req r)`；查订单前 `orderRepository.findByIdAndUserId(id, currentUserId)` 校验归属。
- ❌ 反例：不校验入参直接用;按 id 查订单不校验是否属于当前用户（越权）。
- 理由：输入校验与越权检查是 OWASP 基本要求;越权是高频漏洞。

## 12. 设计模式应用

> 实现业务前应判断是否适用设计模式。下方为常见场景→模式映射，**按需取用，禁止为模式而模式**。
> 判断原则：模式应消除重复/分支、提升扩展性，且带来明确收益;若简单逻辑硬套模式反而增加复杂度，则不套。

### 12.1 策略模式 — 多种算法/策略可互换

**条款：存在多种可互换算法（支付渠道、折扣计算、排序规则等）且用 if-else/switch 分支堆砌时，用策略模式消除分支。**

- ✅ 正例：
  ```java
  public interface PaymentStrategy { void pay(Order order); }
  @Component("alipay") public class AlipayStrategy implements PaymentStrategy { ... }
  @Component("wechat") public class WechatStrategy implements PaymentStrategy { ... }
  // 调用：策略注册到 Map，按 key 取，无分支
  @Service public class PaymentService {
      private final Map<String, PaymentStrategy> strategies;
      public void pay(String channel, Order order) { strategies.get(channel).pay(order); }
  }
  ```
- ❌ 反例：`if ("alipay".equals(channel)) {...} else if ("wechat".equals(channel)) {...} else if (...)` —— 加渠道改一堆分支。
- 理由：开闭原则，新策略加实现不改调用方;分支堆砌是坏味道。

### 12.2 工厂方法/简单工厂 — 复杂对象创建解耦

**条款：对象创建逻辑复杂或需按类型创建且调用方不应感知具体类时，用工厂封装创建。**

- ✅ 正例：`OrderFactory.create(type)` 内部按 type 构造不同 Order 子类，调用方拿 `Order` 抽象。
- ❌ 反例：调用方 `if (type == A) return new OrderA(); else return new OrderB();` 创建逻辑散落多处。
- 理由：集中创建逻辑，调用方依赖抽象，便于扩展与测试。

### 12.3 状态模式 — 状态流转

**条款：对象状态多、状态间行为差异大、满屏 `if (state == X)` 时，用状态模式把状态封装为对象。**

- ✅ 正例：订单状态 `CreatedState`/`PaidState`/`ShippedState` 各自实现 `next()`/`cancel()`，上下文持当前状态委托。
- ❌ 反例：`Order` 里 `if (state == PAID) {...} else if (state == SHIPPED) {...}` 散布各方法。
- 理由：状态行为集中、新增状态加类不改既有;消除状态判断分支。

### 12.4 观察者/发布订阅 — 事件通知解耦

**条款：一个动作需触发多个不相关副作用（下单后通知库存、积分、消息）且主流程不应耦合这些副作用时，用事件解耦。**

- ✅ 正例：Spring `ApplicationEventPublisher`，下单后 `publishEvent(new OrderCreatedEvent(order))`，各监听器 `@EventListener` 处理各自副作用。
- ❌ 反例：`orderService.create()` 里直接 `stockService.deduct(); pointService.add(); msgService.send();` —— 主流程耦合所有副作用，改一个动主流程。
- 理由：主流程只关心核心，副作用解耦可独立增删;注意事件是否异步、是否需事务绑定。

### 12.5 建造者模式 — 逐步构建复杂对象

**条款：对象参数多、可选参数多、需不可变构造时，用建造者链式构建，替代多参构造。**

- ✅ 正例：`User.builder().name("a").email("b").age(1).build();`（Lombok `@Builder`）。
- ❌ 反例：`new User("a", null, "b", null, 1, null, ...)` —— 多参构造，参数位易错、可读性差。
- 理由：链式构建可读、可选参数清晰、对象不可变更安全。

### 12.6 适配器模式 — 统一不一致接口

**条款：需统一多个外部系统的不一致接口为内部统一接口时，用适配器包装。**

- ✅ 正例：`SmsAdapter` 把阿里云/腾讯云短信 SDK 的不同方法适配为统一 `send(to, content)`。
- ❌ 反例：业务代码里 `if (vendor == ALI) aliyunSms.sendSms(...); else tencentSms.SendMessage(...)`。
- 理由：隔离外部差异，业务依赖统一抽象，换供应商不动业务。

### 12.7 装饰器/代理模式 — 横切增强不侵入

**条款：需对资源访问增强（缓存、日志、权限、限流）且不改原逻辑时，用装饰器/代理。**

- ✅ 正例：Spring AOP 切面 `@Around` 给 Service 方法加日志/限流，业务代码零侵入。
- ❌ 反例：每个 Service 方法手动在开头加 `log.info("start"); ... log.info("end");`。
- 理由：横切关注点集中，业务代码保持纯粹;注意层次别过深。

### 12.8 模板方法 — 骨架固定步骤可变

**条款：流程骨架固定、个别步骤子类可变时，用模板方法定义骨架。**

- ✅ 正例：`AbstractImporter.importData()` 定义「读文件→校验→入库→收尾」骨架，子类 override 各步骤。
- ❌ 反例：每个 Importer 类复制粘贴整个流程，仅步骤细节不同。
- 理由：复用骨架，子类只填差异，消除重复。

### 12.9 单例模式 — 全局唯一实例

**条款：Spring 环境用容器单例（Bean 默认 singleton），禁止手写双重检查锁单例;仅非 Spring 管理的纯工具类才考虑单例。**

- ✅ 正例：`@Component`/`@Service` 由 Spring 管理单例。
- ❌ 反例：手写 `class ConfigHolder { private static volatile ConfigHolder inst; ... 双重检查锁 ... }` —— Spring 环境下多此一举、易错。
- 理由：容器单例已托管生命周期;手写单例易出并发问题、与容器冲突。

### 12.10 命令模式 — 请求封装为对象

**条款：请求需排队/记录/可撤销（任务队列、操作日志、撤销栈）时，把请求封装为命令对象。**

- ✅ 正例：`Command` 接口含 `execute()`/`undo()`，操作入队执行、可回放撤销。
- ❌ 反例：直接调用方法，无法记录历史、无法撤销重做。
- 理由：命令对象可持久化、排队、撤销，适合需审计/回滚的场景。

### 12.11 通用反例：禁止滥用

**条款：简单逻辑禁止硬套模式;装饰器/代理层次禁止过深;模式应用须有明确收益，否则保持简单。**

- ✅ 正例：一个简单的两分支判断保持 if-else，不强行策略模式。
- ❌ 反例：两层 if 也套策略 + 工厂 + 建造者，调用链绕半天;装饰器套五层，调试时不知哪层出了问题。
- 理由：模式是工具不是目的;过度设计增加理解与维护成本，违背 KISS。

## 13. 测试约定

> 与 dev-implement TDD 对齐（后端无例外，严格红绿）。

**条款：Service 单测用 Mockito mock 依赖;命名 `should_xxx_when_yyy`;Given-When-Then;覆盖正常 + 边界 + 异常;行覆盖 ≥ 80%。**

- ✅ 正例：
  ```java
  @Test
  void should_throw_BizException_when_user_not_found() {
      given(userRepository.findById(1L)).willReturn(Optional.empty());
      assertThatThrownBy(() -> userService.get(1L))
          .isInstanceOf(BizException.class);
  }
  ```
- ❌ 反例：只测正常路径不测异常;测试名 `test1`;断言 `assertNotNull` 过弱。
- 理由：异常路径是 bug 高发区;命名表达意图;后端 TDD 无例外。
