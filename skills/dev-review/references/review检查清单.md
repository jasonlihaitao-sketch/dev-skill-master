# review 检查清单

> 四维度逐项检查。命中即记入「review报告模板.md」对应章节，标严重度。
> 严重度定义：
> - **blocker**：必须修，不修不能交付（安全漏洞、数据损坏、验收点缺失、核心功能错误）
> - **major**：应修，影响正确性/可维护性/符合度（接口不符、漏测、分层违规）
> - **minor**：建议修（复杂度高、命名欠佳、小重复）
> - **nit**：风格层面（格式、注释）

## 1. 需求与方案符合度

- [ ] 每条 requirement 第 4 章验收标准（AC）都有对应实现
- [ ] 实现的接口签名/请求响应类型与 design 第 4 章一致
- [ ] 数据模型/字段与 design 第 5 章一致（含约束、索引）
- [ ] 类的职责/方法与 design 第 6 章类图一致，无多余承担
- [ ] 核心流程与 design 第 7 章时序图一致
- [ ] 无 design 之外的「自创」接口/数据结构（若有，是否回写了 design）
- [ ] 影响面与 design 第 8 章预估一致，无意外扩散

## 2. 测试覆盖与质量

- [ ] 每个 task 第 3 验收点都有至少一个测试
- [ ] requirement 第 4 章每条 AC 有对应测试
- [ ] 边界用例覆盖（空值、极值、并发、超时）
- [ ] 异常路径覆盖（错误码、失败回退）
- [ ] 测试是 Red 先行（非先实现后补）——查测试与实现的提交/编写顺序
- [ ] 测试不依赖实现内部细节（不过度耦合，便于重构）
- [ ] 无被注释/跳过的测试（`.skip`/`todo`）未说明
- [ ] mock 边界合理，不 mock 被测对象本身

## 3. 代码质量与规范

- [ ] 命名符合 project-context 第 5 章（文件/变量/函数/类）
- [ ] 格式符合 lint 配置（跑 lint 无 error）
- [ ] 分层依赖方向正确，无反向依赖（业务不依赖 API 层等）
- [ ] 圈复杂度合理，无超长函数/深嵌套
- [ ] 无明显重复代码（应提取）
- [ ] 无死代码/注释掉的代码
- [ ] 错误处理一致（不吞异常、错误传播到位）
- [ ] 类型安全（无 `any` 滥用/无隐式 any，按技术栈）

## 4. 安全与性能

- [ ] 输入校验（防注入、防越界、防类型混淆）
- [ ] 认证/授权检查到位（对照 project-context 第 7 章安全约束）
- [ ] 敏感数据不日志输出（密码、token、PII）
- [ ] 密码/凭据哈希存储，非明文
- [ ] 无硬编码密钥/凭证
- [ ] 性能：无循环内重复查库/IO，热点路径无 O(n²)
- [ ] 资源释放（连接/文件/锁）无泄漏
- [ ] 并发安全（共享状态有同步，无竞态）
- [ ] 符合 project-context 第 7 章性能/兼容约束

## 结论判定

- 有任何 blocker → **需修复**
- 仅 major 及以下 → **pass**（附 major 修复建议，由人决定是否立即处理）
- 全清 → **pass**
